Vibe Coding Security: 2.74x More Vulnerabilities and How to Fix It

February 2026 • 11 min read

Here’s the number that should change how you work: AI-generated code contains 2.74 times more security vulnerabilities than human-written code. Not 10% more. Not “slightly elevated risk.” Nearly three times the attack surface, shipping straight into production every time you hit accept.

That finding comes from CodeRabbit’s analysis of 470 GitHub pull requests — 320 AI co-authored, 150 human-only. The AI-generated PRs averaged 10.83 issues per PR compared to 6.45 for human-written code. And the types of issues weren’t trivial style complaints. They were the kinds of vulnerabilities that get your company into headlines: SQL injection, authentication bypasses, hardcoded credentials, and broken input validation.

Meanwhile, 92% of developers are using AI coding tools daily. That’s not going to change. Nobody’s going back to writing everything by hand. So the question isn’t whether to use AI for coding — it’s how to use it without shipping a security disaster.

This post breaks down exactly what’s going wrong, why it’s going wrong, and gives you a concrete 10-point checklist to fix it.

The Vulnerability Landscape: What the Data Actually Shows

Let’s get specific about what “2.74x more vulnerabilities” actually looks like in practice. The CodeRabbit study didn’t just count bugs — it categorized them. And the pattern is damning.

Most Common AI-Generated Vulnerabilities

  • Improper password handling — passwords stored in plaintext, weak hashing algorithms, missing salt, no rate limiting on auth endpoints
  • Input validation failures — unsanitized user input flowing directly into database queries, file paths, or shell commands
  • Authentication bypasses — broken session management, missing token verification, JWT tokens that never expire
  • SQL injection — string concatenation in queries instead of parameterized statements, because the training data is full of Stack Overflow answers from 2014
  • Hardcoded credentials — API keys, database passwords, and secret tokens dropped directly into source files

A separate assessment from December 2025 reinforced these findings. Researchers tested 15 applications built entirely with vibe coding tools and found 69 vulnerabilities across them — an average of 4.6 exploitable security holes per app. These weren’t theoretical. They were working exploits against shipping software.

Perhaps the most unsettling data point comes from academic research on SWE-Agent, one of the more capable autonomous coding agents. Of the solutions it generates, 61% are functionally correct — they pass the tests, they produce the right output. But only 10.5% are actually secure. That means roughly five out of six “working” solutions have a security flaw. The code works. It just also happens to be exploitable.

This is the core problem with vibe coding security: functional correctness and security are almost entirely decoupled. Your code runs, your tests pass, your feature works — and you’re still vulnerable.

Why AI Generates Insecure Code

AI doesn’t generate insecure code because it’s “bad at coding.” It generates insecure code for structural reasons that are worth understanding, because they tell you exactly where to focus your review effort.

Training data reflects the internet, not best practices. Large language models learn from billions of lines of open-source code, Stack Overflow answers, tutorials, and blog posts. Most of that code was never written with security in mind. The quick answer on Stack Overflow that shows you how to build a login form rarely includes rate limiting, CSRF protection, or proper session management. The model learned from those answers, and it reproduces those patterns.

AI optimizes for “works,” not “safe.” When you ask an AI to build a user registration endpoint, its objective function is to produce code that compiles, handles the happy path, and matches the pattern you described. Security is a constraint that was never explicitly stated in the prompt. The model doesn’t think, “What could go wrong here?” It thinks, “What code pattern matches this description?”

No awareness of your threat model. A human developer working on a banking application thinks differently about input validation than one building a personal blog. They know the stakes. They know the attackers. They know the compliance requirements. An AI model has zero context about what your application does, who uses it, or what the consequences of a breach would be. It applies the same generic patterns to a payment processor and a todo list.

Context window limitations hide security-relevant code. Security vulnerabilities are often about what’s happening elsewhere in the codebase. Is that auth middleware actually applied to this route? Does the CORS configuration allow the origin this endpoint is serving? Is the rate limiter upstream of this handler? An AI generating a single function doesn’t have visibility into these cross-cutting concerns, even when the rest of the codebase is in its context window.

The “accept all” culture. This one isn’t the AI’s fault. Vibe coding’s ethos — move fast, trust the output, iterate on vibes — actively discourages the kind of careful, skeptical review that catches security issues. When the culture says “just accept and ship,” nobody’s reading the auth logic line by line.

The 10-Point Security Checklist for Vibe-Coded Projects

You’re not going to stop using AI to write code. You shouldn’t. But you need guardrails. This checklist is designed to be practical — things you can actually do on every project, not aspirational security theater.

1. Never accept authentication or authorization code without manual review. This is the single most important rule. Any code that handles login, session management, password storage, token generation, role checks, or access control gets read by a human, line by line. Period. No exceptions. The AI is statistically worst at exactly this category of code.

2. Run a SAST scanner on every PR. Static Application Security Testing tools like Semgrep, CodeQL, or Snyk catch the low-hanging fruit automatically. Set them up in your CI pipeline so they run on every pull request. They won’t catch everything, but they’ll catch the hardcoded credentials and SQL injection patterns that AI loves to produce. If CodeRabbit’s data tells us anything, it’s that automated scanning pays for itself immediately on AI-generated code.

3. Treat all AI-generated environment and secret handling as wrong until proven right. Any time the AI touches .env files, configuration files, API key management, or secret storage, assume the implementation is insecure. Check that secrets aren’t hardcoded. Check that .env is in .gitignore. Check that API keys aren’t logged. Check that credentials aren’t passed in URL parameters. This takes 60 seconds and prevents the most embarrassing class of vulnerability.

4. Validate every database query. Look for string concatenation in SQL queries. Look for ORM methods that accept raw input. Look for missing parameterization. This is a simple pattern match that takes seconds per query, and it’s the difference between a working app and a data breach. AI models still generate string-concatenated SQL in 2026 because their training data is full of it.

5. Audit input validation at every boundary. Every endpoint that accepts user input — form fields, URL parameters, file uploads, API request bodies — needs validation. Check that the AI actually validates types, lengths, and formats. Check that file uploads are restricted by type and size. Check that URL parameters are sanitized before use. AI-generated code consistently skips or under-implements input validation.

6. Run a dedicated security review agent pass. After your implementation agent finishes, run a separate AI session specifically for security review. Give it a prompt like: “Review this codebase for security vulnerabilities. Focus on authentication, input validation, SQL injection, XSS, CSRF, secrets management, and access control. List every issue with file, line number, and severity.” Using one AI to audit another AI catches a surprising amount.

7. Check dependency versions and known vulnerabilities. AI agents pull in packages without checking whether they’re maintained, up to date, or have known CVEs. Run npm audit, pip audit, or cargo audit after every AI session that modifies your dependency tree. Lock your dependency versions. Don’t accept * or latest as version specifiers.

8. Verify CORS, CSP, and HTTP security headers. AI-generated backend code routinely sets Access-Control-Allow-Origin: * because it’s the path of least resistance during development. Check that CORS is locked down to your actual domains. Check that Content-Security-Policy headers are present. Check that cookies have HttpOnly, Secure, and SameSite flags. These take minutes to verify and prevent entire classes of attack.

9. Test the unhappy paths. AI-generated tests almost exclusively cover the happy path. Write or generate tests that specifically target: What happens with malformed input? What happens with an expired token? What happens when a user tries to access another user’s data? What happens when the request body is 10MB? If your test suite only proves the feature works, it proves nothing about security.

10. Maintain a security-focused CLAUDE.md section. Add explicit security requirements to your project’s memory file. List your authentication approach, your input validation standards, your secrets management strategy, and your known threat model. When every agent session starts by reading these requirements, you get meaningfully better security in the generated output. The AI follows instructions — you just have to give them.

Quick Reference: The Non-Negotiable Three

If you only do three things from this list, do these:

  • Manually review all auth code — the highest-impact, lowest-effort security check you can make
  • Run SAST in CI — catches the mechanical vulnerabilities automatically, on every PR
  • Run a security-focused agent pass — uses AI’s own strength against its weaknesses

Using AI Agents for Security (Not Just Against It)

Here’s the counterintuitive part: AI is simultaneously the cause of and the best available solution to the vibe coding security problem.

Databricks published a paper called “Passing the Security Vibe Check” that makes this point clearly. The same models that generate insecure code can also find insecure code — when given the right instructions and the right framing. The problem isn’t capability. It’s that nobody asks.

Set up a dedicated security review agent. In your development workflow, create a distinct agent session whose only job is security auditing. Don’t mix it with your implementation session. The implementation agent’s job is to make features work. The security agent’s job is to break them. Give the security agent your threat model, your compliance requirements, and explicit instructions to be adversarial.

Use Claude Code to audit your own code. Point Claude Code at your codebase and ask it to find vulnerabilities. Ask it to trace data flow from user input to database query. Ask it to check whether every authenticated route actually verifies the session token. Ask it to list every place a secret is referenced. These are the exact tasks that are tedious for humans and trivial for AI — as long as you actually run them.

Automate security checks as pre-commit hooks. Use an AI agent to generate a pre-commit configuration that checks for common security antipatterns: hardcoded strings that look like secrets, SQL queries built with string concatenation, eval() calls with user input, overly permissive CORS headers. These hooks run before code ever reaches your repository.

The pattern is clear: use AI for implementation, then use AI again — with different instructions and a different mindset — for security review. Two passes, two perspectives, dramatically fewer vulnerabilities.

Key Takeaways

The security problem in AI-generated code is real, measurable, and not going away on its own. But it is manageable — if you build the right habits.

  • AI-generated code has 2.74x more security vulnerabilities than human-written code, based on CodeRabbit’s analysis of 470 GitHub PRs. This isn’t speculation. It’s measured.
  • Functional correctness doesn’t mean security. Only 10.5% of SWE-Agent’s working solutions are actually secure. Your tests passing tells you nothing about whether your auth is exploitable.
  • The vulnerability types are predictable — improper password handling, missing input validation, authentication bypasses, SQL injection, and hardcoded credentials. They’re the same categories every time, which means they’re checklistable.
  • AI generates insecure code for structural reasons: training data bias, optimization for function over safety, no threat model awareness, and context window limits. Understanding why helps you know where to look.
  • The 10-point checklist works because it targets exactly where AI fails. Manual auth review, SAST in CI, security agent passes, dependency auditing, and unhappy-path testing address the specific blind spots in AI-generated code.
  • Use AI to fix what AI broke. A dedicated security review agent, running with adversarial instructions against your implementation agent’s output, catches a significant percentage of the vulnerabilities. Two passes, two perspectives.
  • Stopping isn’t realistic. 92% of developers use AI coding tools daily. The answer is better process, not less AI. Build security into your workflow and it becomes automatic, not burdensome.
