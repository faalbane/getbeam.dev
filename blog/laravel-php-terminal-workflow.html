Laravel Artisan Power User: Organize Your PHP Development Terminal

February 11, 2026 • 9 min read

Laravel dominates PHP development. According to the 2025 JetBrains developer survey, 61% of PHP developers use Laravel as their primary framework. And at the center of every Laravel project is Artisan, the command-line interface that handles everything from spinning up a local server to running database migrations, processing queued jobs, and generating boilerplate code.

The problem is that a real Laravel development session does not live in one terminal. You need php artisan serve running in the background. You need npm run dev for Vite. You need a queue worker. You probably have Tinker open for quick queries. Maybe a test watcher. And now, with AI-assisted coding tools like Claude Code becoming essential to the workflow, you need yet another terminal for your AI agent. That is six terminals minimum, and most developers end up with a tangled mess of tabs with no clear organization.

This guide shows you how to set up a structured Laravel terminal workflow using Beam that keeps every process visible, every command accessible, and your entire development environment restorable with a single click.

The Multi-Terminal Reality of Laravel Development

Let's be honest about what a typical Laravel development session looks like. You open your terminal and start php artisan serve. Then you realize you need Vite for your frontend assets, so you open another tab and run npm run dev. A few minutes later you need to process a queued job, so you open a third tab for php artisan queue:work. Then you want to check a model's attributes quickly, so you open Tinker. Before you know it, you have half a dozen terminals open and you are constantly hunting for the right one.

Here is what is actually running during active Laravel development:

  • php artisan serve — The built-in development server on port 8000
  • npm run dev — Vite dev server with hot module replacement for Blade, Vue, or React assets
  • php artisan queue:work — Processing queued jobs (emails, notifications, file processing)
  • php artisan tinker — The PsySH REPL for inspecting models, running queries, testing logic
  • php artisan test or ./vendor/bin/pest — Your test runner
  • claude — Claude Code for AI-assisted development, refactoring, and code generation

That is six concurrent processes. Each one needs to stay running. Each one produces output you may need to check at any moment. And switching between them by clicking through a flat list of identical-looking tabs is a constant source of friction.

The Beam Workspace Setup for Laravel

The solution is to organize all of these processes into a single named workspace with clearly labeled tabs. In Beam, press ⌘N to create a new workspace and name it after your Laravel project.

Here is the workspace layout that works for most Laravel projects:

Workspace: "MyLaravelApp"

  • Tab 1: Artisan Serve — php artisan serve
  • Tab 2: Vite — npm run dev
  • Tab 3: Queue Worker — php artisan queue:work
  • Tab 4: Tinker — php artisan tinker
  • Tab 5: Tests — php artisan test or ./vendor/bin/pest
  • Tab 6: Claude Code — claude for AI-assisted Laravel development

Press ⌘T to add each new tab. Double-click the tab label to rename it so you can identify every process at a glance. No more guessing which tab is which.

The key insight is that each tab has a dedicated purpose. Your server output never gets mixed up with your test results. Your queue worker logs never scroll past your Tinker session. Everything has a home.

Split Pane Workflows for Laravel

Some processes are closely related and benefit from being visible at the same time. Beam's split panes let you view two terminals side by side within a single tab. Press ⌘⌥⌃T to split the current tab.

Server + Vite Side by Side

Split Tab 1 so you can see php artisan serve and npm run dev output simultaneously. When you make a Blade or Vue component change, you see both the backend response and Vite's HMR update at the same time. This is especially useful when debugging asset compilation errors or tracking down 500 errors that only appear when both servers are running.

# Left pane
php artisan serve

# Right pane
npm run dev

Tinker + Test Runner Side by Side

Split another tab to have Tinker on the left and your test runner on the right. Use Tinker to experiment with Eloquent queries and model relationships, then immediately verify your assumptions by running the relevant test:

# Left pane: Tinker
php artisan tinker
>>> User::where('email', 'like', '%@example.com')->count()
=> 42
>>> Order::with('items')->find(1)->total
=> 129.99

# Right pane: Run specific test
php artisan test --filter=OrderTotalTest

Tip: Use ⌘⌥←→ to switch between workspaces if you are working on multiple Laravel projects simultaneously. Each workspace is completely isolated, so your e-commerce backend workspace will never interfere with your SaaS dashboard workspace.

Laravel Sail Users: Docker Alongside Artisan

If you are using Laravel Sail, your workflow is Docker-based rather than running Artisan directly. Sail wraps everything in Docker Compose, so your commands change slightly but the organizational principle remains the same.

# Start Sail (replaces artisan serve + mysql + redis + etc.)
./vendor/bin/sail up

# Run Artisan commands through Sail
./vendor/bin/sail artisan queue:work
./vendor/bin/sail artisan tinker
./vendor/bin/sail artisan test

# Vite through Sail
./vendor/bin/sail npm run dev

The advantage of the Beam workspace approach is that you can dedicate one tab to the main sail up output, which shows logs from all Docker containers, and then have separate tabs for individual Sail commands. When a Redis connection drops or MySQL crashes, you see it immediately in the Sail tab without hunting through mixed output.

Sail Workspace Layout

  • Tab 1: Sail Up — ./vendor/bin/sail up (all container logs)
  • Tab 2: Vite — ./vendor/bin/sail npm run dev
  • Tab 3: Queue — ./vendor/bin/sail artisan queue:work
  • Tab 4: Tinker — ./vendor/bin/sail artisan tinker
  • Tab 5: Tests — ./vendor/bin/sail artisan test
  • Tab 6: Claude Code — claude

Artisan Command Reference by Workflow

Artisan has hundreds of commands. Here are the ones you will use daily, organized by the workflow phase they belong to.

Development Server & Environment

# Start the development server
php artisan serve

# Start on a custom port
php artisan serve --port=8080

# Clear all caches (config, route, view, application)
php artisan optimize:clear

# Rebuild all caches for performance
php artisan optimize

Database & Migrations

# Run pending migrations
php artisan migrate

# Roll back the last migration batch
php artisan migrate:rollback

# Reset and re-run all migrations + seeders
php artisan migrate:fresh --seed

# Check migration status
php artisan migrate:status

# Generate a new migration
php artisan make:migration create_orders_table

Code Generation

# Generate a model with migration, factory, seeder, and controller
php artisan make:model Product -mfsc

# Generate a form request for validation
php artisan make:request StoreProductRequest

# Generate a resource controller
php artisan make:controller ProductController --resource

# Generate a middleware
php artisan make:middleware EnsureApiKeyIsValid

# Generate an event and listener
php artisan make:event OrderPlaced
php artisan make:listener SendOrderConfirmation --event=OrderPlaced

Queues & Jobs

# Start the queue worker
php artisan queue:work

# Process a single job and stop
php artisan queue:work --once

# Work with a specific queue
php artisan queue:work --queue=emails,default

# Restart all queue workers (after code changes)
php artisan queue:restart

# List failed jobs
php artisan queue:failed

# Retry a failed job
php artisan queue:retry 5

Debugging & Inspection

# List all registered routes
php artisan route:list

# Filter routes by name or URI
php artisan route:list --name=api
php artisan route:list --path=users

# Show application environment info
php artisan about

# Open Tinker REPL
php artisan tinker

Tip: Keep a dedicated Tinker tab open at all times. It is the fastest way to test Eloquent queries, check model relationships, and verify business logic without writing a full test. Use Artisan::call('migrate:status') inside Tinker to check migration state without switching tabs.

Project Memory: Teaching Claude Code Your Laravel Conventions

Claude Code becomes dramatically more useful when it understands your specific Laravel project. Instead of giving it generic instructions every session, use a CLAUDE.md project memory file in your Laravel project root to teach it your codebase conventions once.

Here is an example memory file for a Laravel project:

# Project Memory: MyLaravelApp

## Stack
- Laravel 11, PHP 8.3, MySQL 8, Redis
- Frontend: Blade + Livewire 3, Tailwind CSS
- Testing: Pest PHP with parallel execution

## Architecture
- Domain-driven structure: app/Domain/{Feature}/
- Actions pattern: single-purpose action classes
- Form Requests for all validation
- API resources for all JSON responses

## Models & Relationships
- User hasMany Orders, hasOne Profile
- Order belongsTo User, hasMany OrderItems
- Product belongsToMany Category (pivot: category_product)

## Conventions
- Route model binding everywhere
- Gate-based authorization (no policies)
- Queue all emails via ShouldQueue
- Money stored as integers (cents), cast with MoneyCast
- Dates use Carbon, stored as UTC

## Commands
- php artisan test --parallel (always run parallel)
- php artisan pint (code style before commit)
- ./vendor/bin/pest --filter={test} (run specific test)

With this memory in place, Claude Code knows your models, your patterns, and your conventions. When you ask it to create a new feature, it follows your project's existing architecture instead of defaulting to generic Laravel scaffolding.

Quick Switcher: Jumping Between Laravel Projects

Most Laravel developers work on more than one project. You might have a client's e-commerce site, an internal admin panel, and a personal SaaS product all in various stages of development. Each one gets its own Beam workspace with its own set of tabs.

Press ⌘P to open the Quick Switcher. Type a few characters from the workspace name, tab name, or even the command running in that tab, and Beam fuzzy-matches across everything. Type "tinker" and it finds your Tinker session regardless of which workspace it is in. Type "queue" and you jump straight to the queue worker tab for whichever project you need.

This is transformative when you are context-switching between projects. Instead of mentally tracking which window belongs to which project and which tab is running which command, you just search and jump.

Save Layouts: One-Click Workspace Restoration

Setting up six tabs with the right commands in the right working directories takes a few minutes. You only want to do it once. Beam's layout saving lets you capture your entire workspace configuration and restore it instantly.

  1. Set up your workspace with all six tabs, split panes, and working directories
  2. Press ⌘S to save the layout
  3. Name it "Laravel - MyApp"
  4. Tomorrow, restore the layout and your entire development environment is back exactly as you left it

This is especially powerful if you work on multiple Laravel projects. Save a layout for each one. Monday morning, you open Beam, restore the layout for whichever project you are focusing on, and you are coding within seconds. No setup. No remembering which commands to run. No checking which port is free.

Layout Tip: Multiple Environments

Save separate layouts for different development modes. One layout for full-stack work (server, Vite, queue, Tinker, tests, Claude Code). Another for API-only work (server, queue, tests, Claude Code). Another for frontend-focused work (Vite, tests, Claude Code). Switch between them depending on what you are building that day.

Testing with Pest: Parallel Runner in Its Own Tab

Pest has become the preferred testing framework in the Laravel ecosystem. Its expressive syntax, architectural testing, and parallel execution make it a significant upgrade over PHPUnit. Give it a dedicated tab in your workspace so you can run tests continuously without interrupting anything else.

# Run the full test suite in parallel
php artisan test --parallel

# Run a specific test file
./vendor/bin/pest tests/Feature/OrderTest.php

# Run tests matching a filter
./vendor/bin/pest --filter="can place order"

# Run only tests that failed last time
./vendor/bin/pest --retry

# Run with coverage report
./vendor/bin/pest --coverage --min=80

# Run architectural tests only
./vendor/bin/pest tests/Architecture/

With Pest running in its own tab and Claude Code in another, you get a tight feedback loop. Ask Claude Code to generate a feature, switch to the test tab with ⌘P, run the tests, and if something fails, paste the error back into Claude Code. The entire cycle takes seconds, not minutes.

Tip: Use Pest's --retry flag to re-run only the tests that failed on the last run. Combined with a dedicated test tab, this gives you the fastest possible feedback loop when fixing broken tests.

Bringing It All Together

Laravel development in 2026 involves more concurrent terminal processes than ever before. Between Artisan commands, Vite, queue workers, Tinker, Pest, and AI coding agents, your terminal is the nerve center of your entire workflow. Treating it as an afterthought leads to constant friction and lost context.

With Beam, you get a structured system that scales with your workflow:

  • Named workspaces isolate each Laravel project completely
  • Labeled tabs give every process a permanent, identifiable home
  • Split panes let you monitor related processes side by side
  • Quick Switcher (⌘P) gets you to any terminal in any project instantly
  • Saved layouts (⌘S) restore your entire environment with one click
  • Project memory makes Claude Code understand your specific codebase conventions
  • Undo Close (⌘Z) recovers accidentally closed tabs

Stop hunting for terminals. Start building features.
